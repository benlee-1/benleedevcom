Great Teacher Onigiri: Shipping a Poker Theory App to the App Store

March 2026 10 min read
React 19 Capacitor Supabase RevenueCat Mobile

The Problem Worth Solving

Game Theory Optimal (GTO) poker is the study of unexploitable strategy — the mathematical ideal your decisions should approximate. Most serious players know they should learn it. Almost none actually do, because the resources are terrible.

Solvers like PioSOLVER cost hundreds of dollars and output raw strategy trees that require significant interpretation. Training courses are dense video lectures. Charts are static. None of them close the loop on whether you actually retained anything.

The insight behind Great Teacher Onigiri is simple: the Duolingo model is a genuinely good fit for poker theory. Short feedback loops. Immediate correctness signals. Spaced repetition baked into the curriculum structure. The best poker learners don't memorize charts — they build intuitions through repetition with feedback, and that's exactly what gamification delivers.

So I built it. "Duolingo for GTO poker," with a mascot, a hearts system, Boss Battles, and a Player Profile that tracks your actual leaks.

Architecture Decisions

The key technology choices were driven by a single constraint: this needed to ship on iOS and Android as a native app, while remaining a React web app at its core.

React 19 + Vite

Modern React with fast HMR and excellent Capacitor compatibility. React 19's concurrent features weren't strictly necessary for v1 but the improved Suspense model made handling async auth state cleaner. Vite's build output is what Capacitor wraps — keeping the build pipeline simple was worth a lot.

Capacitor for Native Packaging

Capacitor wraps your web app in a native iOS/Android shell and gives you access to native APIs through plugins. The tradeoff versus React Native is real: the community is smaller, generated native code is fragile, and you'll hit Xcode incompatibilities that take time to diagnose. The upside is that you're writing a web app first — the full Tailwind/React ecosystem works without modification, and the web version is a first-class deployment target.

Supabase

Postgres + auth + Row Level Security in one service. User session data (hand history, XP, metric deltas) needed isolation by user ID, and Supabase's RLS made that clean to enforce at the database layer without middleware. OAuth via Google and Apple is handled by Supabase's auth module, which reduced the auth surface area considerably.

RevenueCat

In-app purchase receipt validation and entitlement sync. The alternative was building the IAP flow directly against StoreKit (iOS) and Google Play Billing, validating receipts server-side, and syncing entitlements into Supabase manually. RevenueCat handles all of that — you get a webhook that fires on subscription events, and your app queries entitlements through the RevenueCat SDK. Significant complexity removed.

The Freemium Hearts Model

Onigiri Pro ($9.99/mo or $49.99/yr) unlocks all five worlds, unlimited hearts, unlimited daily drills, and the full metrics dashboard. Free users get World 1, 5 hearts that regenerate one every 30 minutes, and 10 drills per day.

The design decision I'm most deliberate about: hearts gate velocity, not content visibility. Free users can see that Worlds 2–5 exist. They can see their Player Profile metrics. They just can't burn through lessons at unlimited speed. This means the "aha moment" — completing World 1 and seeing their archetype — happens before they ever hit the paywall. Gating the experience before the hook is a conversion killer.

Building the Curriculum

Five worlds, each covering a pillar of GTO fundamentals:

  • World 1 — Rice Ball Basics: hand rankings, position, pot odds, starting ranges, fold equity
  • World 2 — Postflop Play: continuation betting, board texture, equity realization
  • World 3 — Range Construction: balanced ranges, blockers, polarization
  • World 4 — Multiway Dynamics: multiway pot adjustments, range disadvantage
  • World 5 — Advanced Theory: mixed strategies, indifference, bankroll management

Each lesson is a sequence of slides rendered by a shared LessonViewer component. Slides are static JS objects — a type field (text, visual, quiz, drill) plus content props. This means all lesson content ships with the app bundle: zero backend required for any curriculum data, instant load, easy content updates via PR.

The Practice Hub adds interactive drill modes on top of lessons: a Preflop Trainer (random hand + position against GTO charts, with streak tracking and VPIP/PFR stats), Hand Reading scenarios, Pot Odds visualizations, and Outs & Equity exercises. All drills feed the Player Profile.

The Boss Battle Engine

Each world ends with a Boss Battle: a multi-hand poker session against an AI opponent, with performance feedback after each hand. This is the engineering centerpiece.

Hand Evaluation

The engine evaluates the best 5-card hand from 7 cards (2 hole cards + 5 community cards). All 21 combinations of C(7,2) are evaluated against the hand ranking hierarchy: straight flush, four of a kind, full house, flush, straight, three of a kind, two pair, pair, high card. Kicker logic handles ties. The evaluator runs entirely in the browser — no server round-trip on every street.

AI Opponents

The AI opponents are decision trees, not neural nets. Each opponent has a configurable aggression profile, a positional awareness weight, and a range model that approximates how a real player at their skill tier would construct ranges. Three difficulty tiers: a calling station (over-VPIP, low fold equity), a TAG (tight-aggressive, straightforward), and a GTO-approximator (balanced ranges, position-aware sizing). The opponents are intentionally exploitable in known ways — the game is teaching you to identify and punish those patterns.

Coaching Feedback

After each hand, the engine compares the player's action sequence to the GTO-optimal line for that spot. The delta is translated into plain language: "You called on the flop with 28% equity — you needed 33% to break even. Consider folding or raising here." This feedback loop is the core learning mechanism. Seeing the gap between what you did and what you should have done, in plain English, after every hand, is what builds intuition.

Player Profile and Leak Detection

Every action in every Boss Battle and drill session feeds seven tracked metrics:

  • VPIP — Voluntarily Put money In Pot (preflop call/raise %)
  • PFR — Preflop Raise percentage
  • Aggression Factor — (bets + raises) / calls
  • WTSD — Went To ShowDown rate
  • W$SD — Won money at ShowDown rate
  • 3-Bet % — Re-raise frequency preflop
  • Fold to 3-Bet — How often you fold to a re-raise

These are derived from session hand history stored in Supabase. The leak detection system cross-references metric ranges against known leak archetypes. High VPIP + low aggression factor = calling station. Low VPIP + low PFR = too tight/passive. High W$SD + low WTSD = not getting to showdown enough. The "Top 3 Leaks" panel surfaces the most actionable gaps and links directly to the drill mode that targets each one.

The Player Profile doesn't tell you what to study next — it shows you what your hands have already decided for you. That shift from prescription to diagnosis is what makes it feel like a coach rather than a checklist.

Three Bugs That Taught Me the Most

1. The SPA 404

After deploying to Vercel, navigating directly to /worlds/1 returned a 404. The app worked fine when navigating from the home screen, but any hard refresh or direct URL hit was dead.

Root cause: Vercel serves static files by matching URL paths to files on disk. /worlds/1 isn't a file — it's a client-side route handled by React Router. The server never knew about it.

Fix: vercel.json with a catch-all rewrite: 

{
  "rewrites": [{ "source": "/(.*)", "destination": "/index.html" }]
}

This tells Vercel to serve index.html for any path that doesn't match a real file, letting React Router take over. Lesson: every SPA deployed to a static host needs this configuration. It's easy to miss because local dev servers handle it automatically.

2. The AppDelegate Xcode 16 Build Error

After updating to Xcode 16, the iOS build broke with a cryptic overload resolution error in AppDelegate.swift — a file Capacitor generates, not one I wrote.

Root cause: Xcode 16 introduced stricter protocol conformance checking. Capacitor's generated AppDelegate called a method on a UIApplicationDelegate reference, but Xcode 16 couldn't resolve the overload without an explicit protocol cast.

Fix: adding an explicit cast — (application.delegate as? AppDelegate)?.someMethod() — resolved the ambiguity. The Capacitor team patched this in a subsequent release.

Lesson: generated native code is the most fragile part of a Capacitor project. Before updating Xcode or Capacitor, check the release notes and open GitHub issues. The Capacitor community is small enough that obscure build errors usually have an open thread.

3. The Blank-Screen Race Condition

After the Supabase OAuth callback, the app occasionally showed a blank white screen. Not a crash — no error in the console. Just white.

Root cause: the Supabase auth listener fired and updated global state before the component tree finished mounting. The state update applied, then the initial render wiped it. The auth listener fired again, but the component was already in an inconsistent state.

Fix: redesigned the auth state as an explicit three-state machine — loading | authenticated | unauthenticated — with the component tree gated behind the loading state resolving. Nothing renders until we know the auth state. This eliminated the race entirely.

Lesson: auth state is fundamentally async. Treating it as a boolean (isLoggedIn: true/false) misses the third state — we don't know yet. Model it as a three-state machine and gate your renders accordingly.

What's Next

On the roadmap: a leaderboard for Boss Battle scores, daily challenge mode, a hand replay feature for reviewing past Boss Battle hands, and Worlds 4–5 content completion.
